# Postmortem: Agent 5 Pipeline Abort on Missing Structural Fields

**Date:** 2026-06-26  
**Severity:** High (full pipeline abort, zero output)  
**Status:** Resolved  
**Affected components:** `agents/agent_5_signals.py`  
**Triggered by:** BCG run — 18 active postings → chunked path (2 chunks of ≤15)

---

## Summary

On a Boston Consulting Group run, the pipeline aborted with `FAILED_VALIDATION` at Agent 5. The LLM returned a fully-formed signal analysis for chunk 1/2 — four signal buckets, 13 signals, complete justifications, correct evidence formatting — but omitted three required structural metadata fields: `role_clusters`, `pay_transparency_signal`, and `hiring_velocity`. Pydantic rejected the response. The existing `[PYDANTIC_FALLBACK]` handler logged the error and re-raised, killing the pipeline before chunk 2/2 ever ran and discarding all work completed through Phases 1–4.

The fix patches safe defaults for the three missing fields and retries validation. If validation passes after patching, the corrected result is returned; if it fails for any other reason, the error propagates normally.

## Root Cause Analysis

### The chunking threshold and what triggered it

`CHUNK_SIZE = 15`. BCG had 18 active postings → two chunks: 15 active + 7 stale in chunk 1, 3 active in chunk 2. This was the first run to cross the chunk boundary. All prior runs (Akamai, Lumenalta, Workday) had ≤15 active postings and executed the single-chunk path successfully.

### Why the LLM omitted the fields

Agent 5 passes `InsideScoopOutput.model_json_schema()` as the tool input schema. The schema is generated by Pydantic and lists all 11 required fields. The LLM honored the schema for the 8 narrative-heavy fields (`growth_cues`, `ambiguous_cues`, `stability_cues`, `red_flags`, `red_flags_note`, `org_focus`, `org_focus_justification`, `career_ceiling`) and skipped the 3 structural metadata fields (`role_clusters`, `pay_transparency_signal`, `hiring_velocity`).

This is likely a combination of two factors:

1. **Token budget pressure.** Chunk 1 was large: 22 postings (15 active + 7 stale), each with title, dept, location, source, URL, and classification. The signal analysis the LLM produced was dense — 4 growth cues, 4 ambiguous cues, 3 stability cues, 4 red flags, each with multi-sentence evidence strings citing specific URLs. The three missing fields are listed last in the schema. The LLM appears to have run out of token budget (3000 max_tokens) before completing the output.

2. **Schema field ordering.** Pydantic emits fields in declaration order. The three missing fields are declared last in `InsideScoopOutput` (lines 224–226 in `models.py`). The LLM did not emit them — consistent with truncation at the end of the output, not selective field skipping.

Neither factor was visible in prior runs because single-chunk inputs are smaller and the LLM completed all fields without hitting the budget ceiling.

### Why the fallback made it worse

The `[PYDANTIC_FALLBACK]` handler in `_call_llm` was written for the Agent 3 pattern: when the LLM returns a **malformed** response (wrong enum value, wrong type), the correct recovery is to emit a safe default. For Agent 5, the handler was copied with the same re-raise behavior — except Agent 5's failure mode is different. The LLM's response was semantically correct and schema-conforming for 8 of 11 fields. Re-raising discarded work that was recoverable.

**Pre-patch code path:**

```python
except ValidationError as e:
    print(f"[PYDANTIC_FALLBACK] agent_5 {chunk_label} — ...")
    raise  # ← kills the pipeline; discards all good signal output
```

The `raise` propagates through the chunked loop in `run_agent_5`, up through `run_osint_pipeline`'s outer `except ValidationError` block, setting `state.status = "FAILED_VALIDATION"`. Chunk 2/2 never executes. All signal work from Phases 1–4 is saved to disk but the Inside Scoop section is empty.

## Fix

The fallback now patches safe defaults for the three structurally-missing fields and retries validation. If the patch produces a valid `InsideScoopOutput`, it is returned and the run continues. If validation still fails (a genuine malformed response, not just truncation), the error propagates as before.

```python
except ValidationError as e:
    print(f"[PYDANTIC_FALLBACK] agent_5 {chunk_label} — ...")
    patched = dict(block.input)
    patched.setdefault("role_clusters", [])
    patched.setdefault("pay_transparency_signal", PayTransparency.ABSENT.value)
    patched.setdefault("hiring_velocity", HiringVelocity.MEDIUM.value)
    if not patched.get("red_flags"):
        patched.setdefault("red_flags_note", "No red flags detected in this pass.")
    print(f"[PYDANTIC_FALLBACK] retrying with patched defaults for {chunk_label}...")
    return InsideScoopOutput(**patched)
```

**Default values rationale:**

| Field | Default | Rationale |
|---|---|---|
| `role_clusters` | `[]` | Empty list is schema-valid. `_merge_chunks` handles empty cluster maps without error. In a chunked run, the other chunk may populate clusters via the merge step. |
| `pay_transparency_signal` | `ABSENT` | Conservative. The system prompt doesn't explicitly instruct the LLM to check salary ranges in BCG postings; ABSENT reflects the most common true state. |
| `hiring_velocity` | `MEDIUM` | Neutral. In a multi-chunk run, the second chunk's value participates in majority-vote merge and can override this default. In a single-chunk run where this triggers, MEDIUM is the least-wrong neutral signal. |

**What is not patched:** `growth_cues`, `ambiguous_cues`, `stability_cues`, `red_flags`, `org_focus`, `career_ceiling`, and all narrative fields. If those are missing or malformed, the exception propagates unchanged — those fields require actual LLM judgment and cannot be defaulted safely.

## Lessons Learned

**1. Failure modes differ between agents; fallback strategy can't be copy-pasted.**  
Agent 3's fallback re-raises because `DATE_UNKNOWN` from a malformed response is an acceptable safe default — the classification is just one field. Agent 5's fallback re-raises the same way, but the output is 11 fields of multi-signal analysis. The right recovery depends on whether the LLM's failure was structural (missing fields) or semantic (wrong values). Don't assume the same handler works for both.

**2. `max_tokens` is a hard ceiling, not a soft suggestion.**  
Chunk 1 produced a dense, evidence-rich output. The LLM ran out of budget before completing the struct tail. The fix addresses the symptom (patch missing fields), but the root mechanism is token exhaustion. The cleaner long-term fix is to raise `max_tokens` from 3000 to 4000–5000 for Agent 5, or to split the system prompt's structural fields into a second call after signals are extracted.

**3. The chunked path was never exercised before shipping.**  
`CHUNK_SIZE = 15` exists, the chunking logic in `run_agent_5` exists, and `_merge_chunks` exists — but every target run in M1 and M2 had ≤15 active postings. The chunked path was dead code in production until BCG. A test fixture with 16–20 active postings would have caught this before the first real run.

**4. Fields declared last in a Pydantic model are highest-truncation-risk in tool use.**  
The LLM populates tool input in schema order. Required fields at the bottom of the schema are the first to be omitted under token pressure. For critical fields that must survive truncation, either declare them early or add explicit prompt instructions calling them out.
